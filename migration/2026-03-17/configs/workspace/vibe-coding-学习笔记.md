# Vibe Coding 提示词训练逻辑深度解析

> 📚 基于 GitHub 两个核心项目的学习总结  
> 📅 2026-03-08 | 学习来源：liyupi/ai-guide + KhazP/vibe-coding-prompt-template

---

## 🎯 核心理念

**Vibe Coding = 结构化提示词 + 分阶段工作流 + AI 代理协作**

### 关键思维转变

| 传统提示词思维 | Vibe Coding 对话思维 |
|---------------|-------------------|
| 一次性写出完美提示词 | 迭代式对话逐步明确需求 |
| 单向指令输入 | 双向探索交流 |
| 期待魔法般完美答案 | 接受渐进式完善过程 |
| 认知负担重 | 边聊边想，降低门槛 |

---

## 📐 5 阶段结构化工作流

```
💡 创意 → 📊 市场调研 → 📋 PRD → 🏗️ 技术设计 → 🤖 AGENTS.md → 🚀 MVP
```

### 阶段 1: 深度调研 (20-30 分钟)

**目标**: 验证创意是否值得做，避免盲目开发

**提示词设计逻辑**:
```markdown
1. 首先判断用户技术背景 (Vibe  coder/Developer/中间)
2. 根据背景选择不同提问路径
3. 逐个问题询问（不是一次性抛出所有问题）
4. 执行"验证回声"确认理解正确
5. 生成综合调研报告
```

**关键技巧**:
- ✅ 开启联网搜索确保数据最新
- ✅ 区分事实数据和模型知识
- ✅ 标注信息来源 URL 和访问日期
- ✅ 保持单一持续会话避免上下文丢失

**示例提问路径** (针对 Vibe-coder):
```
Q1: "你的 App 创意是什么？用跟朋友解释的方式描述——它解决什么问题？"
Q2: "谁最需要这个？描述理想用户（如'忙碌的父母'、'小企业主'）"
Q3: "现有解决方案有哪些？列出竞品或替代方案"
Q4: "用户为什么选择你的 App？独特优势是什么？"
Q5: "上线必需的 3 个核心功能？只要 essentials！"
Q6: "用户使用场景——手机 App、网站、还是两者？"
Q7: "时间线？几天、几周、还是几个月上线？"
Q8: "预算现实检查：能花钱买工具还是需要全免费？"
```

---

### 阶段 2: 产品需求文档 PRD (15-20 分钟)

**目标**: 将模糊创意转化为可执行的 MVP 范围

**提示词设计逻辑**:
```markdown
输入：阶段 1 的调研报告
处理：AI 澄清模糊需求 → 优先级排序 → 功能边界定义
输出：PRD-[项目名]-MVP.md
```

**核心结构**:
```markdown
# PRD: [App Name]

## 1. 产品概述
- 一句话价值主张
- 目标用户画像
- 核心痛点

## 2. MVP 功能列表
### P0 - 必须有（上线前必须完成）
- [功能 1]
- [功能 2]

### P1 - 应该有（可以让 MVP 更好）
- [功能 3]

### P2 - 可以有（后续迭代）
- [功能 4]

## 3. 用户故事
- 作为 [用户类型]，我想要 [功能]，以便 [价值]

## 4. 成功指标
- [可量化的指标 1]
- [可量化的指标 2]

## 5. 不做清单（明确排除范围）
- [明确不做 1]
- [明确不做 2]
```

---

### 阶段 3: 技术设计 (15-20 分钟)

**目标**: 选择可落地、符合预算的技术栈

**提示词设计逻辑**:
```markdown
输入：PRD 文档
处理：AI 询问预算/时间/复杂度 → 提供多方案权衡 → 共同决策
输出：TechDesign-[项目名]-MVP.md
```

**技术选型矩阵**:

| 场景 | 推荐工具 | 特点 |
|------|---------|------|
| 快速原型 | Lovable | 包含 Agent/Plan 模式、DB、Auth |
| 生产级前端 | v0 | Vercel 原生，精确 Next.js/React 组件 |
| 学习/沙盒 | Cursor | 动态上下文管理 |
| 复杂逻辑 | Claude Code | 多代理团队协作 |
| 预算有限 | Gemini CLI + VS Code | 免费开源 |

**关键决策点**:
- 全代码 vs 零代码/低代码
- 单体架构 vs 微服务
- 自托管 vs 云服务
- SQL vs NoSQL
- 认证方案选择

---

### 阶段 4: 生成代理文件 (1-2 分钟)

**目标**: 创建 AI 编程代理的工作指令文档

**提示词逻辑**:
```
"读取 part4-notes-for-agent.md，跟随指示设置工作区"
```

**AGENTS.md 核心结构**:
```markdown
# AGENTS.md — Master Plan for [App Name]

## 项目概述与技术栈
**App**: [名称]
**概述**: [一段话描述核心价值]
**技术栈**: [Next.js, React, Node, PostgreSQL]
**关键约束**: [移动端优先、多租户、严格 TypeScript]

## 设置与命令
- **安装**: `pnpm install`
- **开发**: `npm run dev`
- **测试**: `npm test`
- **构建**: `npm run build`

## 受保护区域（禁止修改）
- 基础设施配置
- 数据库迁移文件
- 第三方集成配置

## 编码规范
- 强制执行 ESLint/Prettier
- 基于功能的目录结构
- 严格 TypeScript（禁止 any）
- 新代码必须有测试

## 代理行为准则
1. **先计划后执行**: 改多文件前必须提计划
2. **重构优于重写**: 渐进式重构而非大段重写
3. **上下文压缩**: 写 MEMORY.md 而非堆砌历史
4. **迭代验证**: 每次改动后运行测试
5. **团队协调**: 多代理时需 Team Lead 审批
```

---

### 阶段 5: 构建 MVP (1-3 小时)

**目标**: 小步迭代，快速交付可用版本

**提示词逻辑**:
```
"读取 AGENTS.md，提出阶段 1 计划，等待审批后逐步构建"
```

**开发循环**:
```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│   📝 计划    │───>│   ⚡ 执行    │───>│   🔍 验证    │
│  (审批通过)  │    │  (一个功能)  │    │  (测试)      │
└──────────────┘    └──────────────┘    └──────────────┘
       ▲                                      │
       └──────────────────────────────────────┘
              迭代直到 MVP 完成
```

**关键实践**:
- ✅ 每个功能点后暂停解释改动
- ✅ 运行测试通过再继续
- ✅ 遇到错误先让 AI 诊断再动手
- ✅ 定期提交 Git 保留回滚点

## 🔑 对话工程核心技巧

### 技巧 1: 从大到小，逐步细化

❌ 错误示范:
```
帮我做一个支持 Markdown、代码高亮、评论、点赞、分类、标签、
搜索、RSS 订阅的博客系统
```

✅ 正确做法:
```
1. 我想做个简单的博客系统，用户能发布和查看文章
2. 文章需要支持 Markdown 格式
3. 能不能加上代码高亮功能
4. 我还想要评论功能
```

---

### 技巧 2: 具体而非抽象

❌ 不好: `做一个好看的按钮`  
✅ 好: `做一个圆角按钮，蓝色背景 (#3B82F6)，白色文字，padding 上下 12px 左右 24px，悬停时背景变深 (#2563EB)`

❌ 不好: `加一个用户友好的错误提示`  
✅ 好: `当邮箱格式错误时，在输入框下方显示红色文字"请输入有效的邮箱地址"`

---

### 技巧 3: 提供参考和示例

**文字参考**:
```
- 类似 GitHub 的个人主页布局：左边用户信息，右边活动时间线
- 按钮样式参考 Stripe：简洁现代，微妙阴影
- 表单验证参考 Airbnb：实时验证，错误在输入框下方
```

**图片参考** (支持多模态的 AI):
```
- 【上传截图】请参考这个布局设计我的页面
- 【上传截图】我的页面移动端显示不正常，怎么修复？
```

---

### 技巧 4: 分步骤提问

实现登录功能示例:
```
1. 先帮我创建登录表单：邮箱 + 密码输入框 + 登录按钮
2. 添加客户端验证：邮箱格式、密码至少 6 位
3. 点击按钮发送 POST 到 /api/login，携带邮箱和密码
4. 登录成功跳转首页，失败显示错误信息
```

---

### 技巧 5: 角色赋予

```markdown
你是一位资深的 React 前端工程师，专注于可访问性和性能优化。
请用专业的方式实现这个功能，并解释关键决策。
```

## 🧠 上下文管理策略

### 问题：上下文过载

当对话过长时，AI 会"遗忘"早期内容或响应变慢

### 解决方案

**1. 文件化记忆**
```markdown
- 让 AI 将关键决策写入 MEMORY.md 或 spec.md
- 而不是堆砌在聊天记录中
```

**2. 会话压缩**
```markdown
- 使用工具内置压缩功能（如 Claude 的/compact）
- 而非硬重置开启新会话
```

**3. 交接文档**
```markdown
必须切换会话时，创建 handoff.md:
- 项目摘要
- 最新决策
- 待解决问题
- 下一步计划
```

## 🛡️ 幻觉与死循环处理

### 识别幻觉信号
- AI 自信地引用不存在的 API
- 生成的代码运行时报错
- 反复声称"已修复"但问题依旧

### 应对策略

**1. 要求引用来源**
```
你提到的这个 API 有官方文档链接吗？请提供 URL
```

**2. 启用联网搜索**
```
请开启联网搜索验证这个信息是否准确
```

**3. 强制测试验证**
```
不要只说已修复，请运行测试展示结果
```

**4. 跳出死循环**
```
我们卡住了。让我重新描述问题...
或者：我们换个方法，不用 X 方案，试试 Y 方案
```

## 📊 模型选择策略

| 策略 | 推荐模型 | 适用场景 | 速度 |
|------|---------|---------|------|
| 速度优先 | Gemini Flash, Claude Sonnet | 快速原型、广泛迭代 | 高 |
| 平衡型 | Claude Sonnet, Gemini Pro | 日常编码、调试、规划 | 中高 |
| 深度优先 | Claude Opus, Gemini Pro | 深度推理、复杂重构 | 中 |

**建议**: 使用模型族而非具体版本号（如"Sonnet"而非"Sonnet-3.5"），避免模型迭代导致策略失效

## 💡 关键洞察

1. **提示词不是魔法咒语**，而是结构化沟通工具
2. **分阶段隔离**比单一巨型提示词更有效
3. **文档化决策**比依赖聊天记录更可靠
4. **小步迭代**比一次性完美主义更实际
5. **验证优先**比盲目信任 AI 输出更安全
